1. DESIGN PHILOSOPHY
================================================================================

The Geareo developer made three key design decisions:

(1) DO NOT WRITE A CUSTOM PHYSICS ENGINE.
    - Let Unity/PhysX handle Rigidbody dynamics, constraint solving,
      force integration, and collision response.
    - The custom code only handles gear-specific logic: tooth contact
      detection, gear scanning/pairing, and anchor geometry.

(2) DO NOT COMPUTE THE GEAR RATIO.
    - Instead of calculating the ratio mathematically (rA/rB) and
      directly setting transforms, let the ratio EMERGE from geometry.
    - A ConfigurableJoint with all 6 DOF locked forces two anchored
      points on the gear circumferences to stay together. Because the
      radii differ, the angular displacements differ proportionally.
    - The ratio "falls out" of the physics solver — no math needed.

(3) SEPARATE THE SIMULATION FROM THE RENDERING.
    - Parts simulate in a DEDICATED PhysicsScene, separate from the
      default scene (UI, camera, player, etc.).
    - This gives deterministic behaviour, configurable step rates,
      and isolation from gameplay physics.

This approach is elegant because it offloads the hard work (constraint
solving, torque propagation, mass/inertia handling) to PhysX, while
keeping the gear-specific logic (which PhysX doesn't understand, since
it doesn't have a native "gear joint") as a thin layer on top.

2. THE CORE TRICK — ConfigurableJoint as a Gear Constraint
================================================================================

A standard ConfigurableJoint connects two Rigidbody bodies at specific
anchor points and locks degrees of freedom between them.

For gears, the developer:

  (a) Locks ALL 6 degrees of freedom (3 linear + 3 angular)
      → The two bodies are forced to maintain a fixed relative pose.

  (b) Places the anchor on the PITCH CIRCLE (circumference) of each gear,
      NOT at the center.
      → anchor on Gear A is at (centerA + radiusA * directionToContact)
      → connectedAnchor on Gear B is at (centerB + radiusB * oppositeDirection)

  (c) Recalculates the anchor positions EVERY physics step.
      → As gears rotate, the "contact teeth" change.
      → The anchors move to the current contact point each frame.

  (d) Never explicitly calculates or sets the gear ratio.
      → When Gear A's tooth moves by arc length s, the locked joint
        forces Gear B's corresponding tooth to move by the same arc
        length s. Since radiusB ≠ radiusA, the angular displacement
        is different: θB = s / rB  vs  θA = s / rA.
      → Ratio = θA / θB = rB / rA.

This is the entire mechanism. The beauty is that:
  - There is NO counter-rotation calculation
  - There is NO ratio multiplication
  - There is NO angular velocity setting
  - PhysX does ALL the force/torque/constraint work

The only custom math is:
  - Finding which tooth is closest to the contact point (ToothToPos)
  - Checking if two gears are close enough to mesh (HasContact)
  - Choosing sensible pairs (GearScanner dispatch table)

4. PER-FRAME SIMULATION PIPELINE
================================================================================

The main simulation loop runs in ProjectSim.FixedUpdate():

  ProjectSim.FixedUpdate()
  │
  ├─ Read inputs (from HubSim via HubReceiver on parts)
  │
  ├─ SimTick() on all ISimTick parts
  │    └─ BearingPartSim.SimTick()
  │         └─ HingeJoint.motor.targetVelocity = inputValue
  │
  ├─ SimPrePhysicsUpdate() on all ISimPrePhysicsUpdate parts
  │    │
  │    ├─ SpurGearPartSim.SimPrePhysicsUpdate()
  │    │    ├─ Physics.OverlapSphereNonAlloc(center, radius + padding, cols, layerMask)
  │    │    └─ GearController.Scan(cols, count)
  │    │         ├─ For each hit collider: GetComponentInParent<ISimPart>() → IGear
  │    │         ├─ GearScanner.GetScanner(typeA, typeB)
  │    │         ├─ scanner.CheckContact(a, b) → GearContact
  │    │         ├─ If hasContact && !alreadyConnected:
  │    │         │    scanner.CreateJoint(a, b, contact) → RadialGearJoint
  │    │         │    RadialGearJoint constructor:
  │    │         │      • AddComponent<ConfigurableJoint>
  │    │         │      • Lock all 6 DOF
  │    │         │      • Set anchors at current tooth positions
  │    │         │      • Set connectedBody = other gear's Rigidbody
  │    │         └─ If !hasContact && wasConnected:
  │    │              joint.Destroy(), remove from connections dict
  │    │
  │    └─ (similar for Bevel, Ring, Linear)
  │
  ├─ Physics.Simulate(physicsScene, Time.fixedDeltaTime)
  │    └─ PhysX resolves all ConfigurableJoint constraints:
  │         When the motor rotates gear A, the locked joint between
  │         A and B forces B to rotate proportionally based on the
  │         anchor positions and radii.
  │
  ├─ SimPhysicsUpdate() on all ISimPhysicsUpdate parts
  │    └─ GearController.Update()
  │         └─ For each GearConn: joint.Update()
  │              └─ RadialGearJoint.Update()
  │                   └─ Calculate(force=false)
  │                        Recalculates anchors from CURRENT rotation
  │                        for the NEXT physics step.
  │
  └─ SimPostPhysicsUpdate() + Transform sync

The critical timing: anchors are recalculated AFTER physics, so the
next physics step uses the updated contact teeth. This creates the
effect of continuous rolling without slip.

5. STEP 1 — NEIGHBOR DETECTION (Physics.OverlapSphereNonAlloc)
================================================================================

  Radial gears (spur/bevel/ring):
    scanRadius = gearRadius + simPadding
    Physics.OverlapSphereNonAlloc(centerOfMass, scanRadius, colliderBuffer, layerMask)

  Linear gears (racks):
    Physics.OverlapBoxNonAlloc(center, halfExtents, colliderBuffer, orientation, layerMask)

  Why OverlapSphere and not OverlapCapsule?
    - Gears are flat cylinders; the sphere centered on the COM with
      radius = gearRadius + padding covers the entire gear volume.
    - Simpler and faster than a capsule query.
    - padding (simPadding from GearModule config) ensures gears that
      are ALMOST touching are included in the scan.

  The collider buffer is a STATIC array reused across frames to avoid
  allocation. After scanning, the buffer is cleared with Array.Clear.

  For each hit collider, GetComponentInParent<ISimPart>() is called to
  find the owning gear. If it implements IGear, it's a valid neighbor.

6. STEP 2 — CONTACT VERIFICATION (HasContact / CheckContact)
================================================================================

  Once neighbors are found, the scanner's CheckContact() method verifies
  whether two gears actually mesh (are at the correct distance for
  tooth engagement).

  SpurSpurGearScanner.CheckContact(a, b) calls HasContact(a, b):

    Input: two IGear references (a, b)
    Output: GearContact { bool hasContact; bool inverted; float distance; }

  Algorithm (reconstructed from CallAnalysis attributes):

    let posA = a.transform.position
    let posB = b.transform.position
    let upA  = a.transform.up          // rotation axis
    let upB  = b.transform.up
    let rA   = a.gearRadius
    let rB   = b.gearRadius

    // Center-to-center distance
    let direction = normalize(posB - posA)
    let centerDist = |posB - posA|

    // Theoretical contact distance
    let contactDist = rA + rB

    // Check proximity
    let proximity = |centerDist - contactDist|

    // Get tooth-tip positions (radius direction from center toward other gear)
    let toothTipA = a.transform.TransformDirection(Vector3(rA, 0, 0))
    let toothTipB = b.transform.TransformDirection(Vector3(rB, 0, 0))

    // Project tooth tips onto the line between centers
    // Uses ProjectPointToSegmentDistance to find how close the
    // tooth tips are to the ideal contact line
    let projDist = ProjectPointToSegmentDistance(toothTipA, posA, posB)

    // Compare against attachDistance threshold from GearModule
    let attachDist = a.gearModule.attachDistance
    if projDist <= attachDist:
        // Gears are meshing
        // Determine if rotation directions are inverted
        // (spur gears always inverse; some bevel/ring combos don't)
        result.hasContact = true
        result.inverted = (type supports inversion)
        result.distance = projDist
    else:
        result.hasContact = false

  ProjectPointToSegmentDistance(P, A, B):
    // Standard 3D point-to-line-segment distance
    let AB = B - A
    let AP = P - A
    let t = clamp(dot(AP, AB) / |AB|², 0, 1)
    let closest = A + t * AB
    return |P - closest|

  This is the ONLY place where custom contact math is performed.
  Everything else is delegated to Unity/PhysX.

================================================================================
7. STEP 3 — JOINT CREATION (RadialGearJoint / GearSimJoint)
================================================================================

  When CheckContact returns hasContact = true and no joint exists yet:

  RadialGearJoint constructor:

    (1) Create ConfigurableJoint component on a host GameObject
        → joint = gameObject.AddComponent<ConfigurableJoint>()

    (2) Configure the joint:
        → joint.connectedBody = other.gearRigidbody
        → joint.autoConfigureConnectedAnchor = false
        → joint.enableCollision = false
        → joint.enablePreprocessing = false

    (3) Lock ALL 6 degrees of freedom:
        → joint.xMotion  = ConfigurableJointMotion.Locked
        → joint.yMotion  = ConfigurableJointMotion.Locked
        → joint.zMotion  = ConfigurableJointMotion.Locked
        → joint.angularXMotion = ConfigurableJointMotion.Locked
        → joint.angularYMotion = ConfigurableJointMotion.Locked
        → joint.angularZMotion = ConfigurableJointMotion.Locked

    (4) Set the joint axis:
        → joint.secondaryAxis = InverseTransformDirection(gear.up)
        → (axis is set later in Calculate)

    (5) Calculate initial anchor positions (see Step 4):
        → This sets joint.anchor and joint.connectedAnchor

  IMPORTANT: Locking angular motions means the relative ORIENTATION
  of the two bodies is fixed. However, because anchors are on the
  circumference (not centers) and updated every step, the effective
  motion allows the gears to "roll" against each other. See Section 10
  for the geometric proof.

8. STEP 4 — ANCHOR PLACEMENT (ToothToPos)
================================================================================

  The anchor for each gear is a point on its PITCH CIRCLE at the
  tooth closest to the contact point.

  To find the tooth index:
    FindContactTooth(gear, other):
      contactPoint = gear.position + normalize(other.position - gear.position) * gear.radius
      // This is the point on gear's pitch circle closest to the other gear

      // Convert to angle around the gear's axis
      dir = contactPoint - gear.position
      projected = ProjectOnPlane(dir, gear.up)
      angle = SignedAngle(gear.right, projected, gear.up)
      toothIndex = round(angle / (360° / gearTeeth)) % gearTeeth

  To compute the world position of a specific tooth:
    ToothToPos(gear, toothIndex):
      anglePerTooth = 360° / gearTeeth
      toothAngle = toothIndex * anglePerTooth

      // Combine gear's world rotation with the tooth's local angle
      rotatedDir = gear.rotation * AngleAxis(toothAngle, gear.up) * Vector3.right
      worldPos = gear.position + rotatedDir * gearRadius

      return worldPos

  The ConfigurableJoint anchor is set from the world position:
    joint.anchor = gear.transform.InverseTransformPoint(worldAnchorPos)
    joint.connectedAnchor = other.transform.InverseTransformPoint(otherWorldAnchorPos)

  These are LOCAL positions relative to each Rigidbody's center.
  PhysX uses them to determine where the constraint is applied.

9. STEP 5 — ANCHOR UPDATE (Calculate / UpdateAnchors)
================================================================================

  Every physics step (in SimPhysicsUpdate / GearController.Update):

    RadialGearJoint.Update() → Calculate(force=false)

    (1) Read current gear positions and rotations from transforms
        (these were just updated by PhysX in the previous Physics.Simulate)

    (2) Recalculate the contact tooth indices:
        gearTooth  = FindContactTooth(gear, other)
        otherTooth = FindContactTooth(other, gear)

    (3) Compute world-space anchor positions:
        anchorA = ToothToPos(gear, gearTooth)
        anchorB = ToothToPos(other, otherTooth)

    (4) Convert to local space for the ConfigurableJoint:
        joint.anchor         = gear.transform.InverseTransformPoint(anchorA)
        joint.connectedAnchor = other.transform.InverseTransformPoint(anchorB)

    (5) Recalculate the joint axis:
        axisDirection = normalize(other.position - gear.position)
        rotatedAxis = Quaternion.AngleAxis(angle, gear.up) * axisDirection
        joint.axis = gear.transform.InverseTransformDirection(rotatedAxis)

    (6) Set linear limit to zero (no linear play):
        joint.linearLimit.limit = 0f

  The anchor positions now represent the CURRENT tooth contact.
  In the NEXT physics step, PhysX will maintain these two points
  coincident, effectively rolling the gears.

  Because positions are read from transforms AFTER PhysX integration,
  and new anchors are set BEFORE the next integration, the cycle is:
    PhysX step → positions update → new anchors → PhysX step → ...

10. WHY THE GEAR RATIO EMERGES — Geometric Proof
================================================================================

  Theorem: When two points on circles of different radii are forced
  to remain coincident, the angular displacements obey the gear ratio.

  Setup:
    - Gear A has radius rA, rotates around axis uA, at position pA
    - Gear B has radius rB, rotates around axis uB, at position pB
    - The ConfigurableJoint locks ALL relative motion between anchorA
      and anchorB (both linear and angular)

  At the start of a physics step:
    - anchorA is at world position W positioned on A's pitch circle
    - anchorB is at the same world position W positioned on B's pitch circle
    - The 6 DOF lock maintains: anchorA(t) = anchorB(t) for all t during the step

  During the physics step:
    - Gear A rotates by angle θA around uA
    - The anchor point on A moves along an ARC of length sA = rA * |θA|
    - Because the joint locks the anchors together, the anchor on B
      must move through the SAME arc length in world space
    - Therefore: sB = sA
    - But sB = rB * |θB|, and sA = rA * |θA|
    - So rB * |θB| = rA * |θA|
    - Hence |θB| = (rA / rB) * |θA|
    - The sign is opposite (gears counter-rotate):
      θB / θA = -(rA / rB) = -(teethA / teethB)

  WHERE THE RATIO COMES FROM:
    - It is NOT stored in any variable
    - It is NOT calculated by any line of code
    - It is NOT enforced by any script
    - It is a CONSEQUENCE of:
        (a) The anchors being on the circumference (not the center)
        (b) The ConfigurableJoint locking all 6 DOF
        (c) PhysX enforcing the rigid-body constraint

  The developer exploited the fact that PhysX is ALREADY a perfect
  constraint solver. By placing the constraint at the right point
  (the pitch circle), the solver naturally enforces the gear ratio.

11. POWER INPUT — Motor System (BearingPartSim / HingeJoint)
================================================================================

  In Geareo, the "motor" is a BearingPartSim — a part that acts as
  a bearing with a driven HingeJoint. This is what makes gears spin.

  BearingPartSim ISimTick() (called before physics each step):
    (1) Read input signal from HubReceiver
        → float inputValue = hubReceiver.GetValue(key)

    (2) Map input to motor velocity via AnimationCurve/mapping
        → float targetVelocity = mapInputToVelocity(inputValue)

    (3) Apply to HingeJoint motor
        → hingeJoint.motor.targetVelocity = targetVelocity
        → hingeJoint.motor.force = maxForce
        → hingeJoint.useMotor = true

  The HingeJoint's axis is aligned with the gear's up vector:
    → hingeJoint.anchor = centerOfMassRelative
    → hingeJoint.axis = transform.InverseTransformDirection(up)

  When the HingeJoint motor spins, the gear's Rigidbody rotates.
  The ConfigurableJoint between this gear and its neighbors then
  propagates the rotation through the gear train (via the locked
  DOF + tooth anchor mechanism described above).

  Power propagates through the GEAR TRAIN:
    Motor → HingeJoint → Rigidbody of gear A → ConfigurableJoint AB
      → Rigidbody of gear B → ConfigurableJoint BC → Rigidbody of gear C
      → ...

  Each ConfigurableJoint enforces the geometric ratio for its pair,
  so the overall train ratio is the product of all pair ratios.

  Optional: BearingPartSim also supports DETENT positions (snap-to-grid)
    - Uses a SECOND ConfigurableJoint with a limit spring
    - The limit spring provides a restoring force near detent positions
    - Snapping.Snap(angle, step) calculates the nearest detent

12. OLD SYSTEM vs NEW SYSTEM
================================================================================

  The codebase contains TWO gear simulation systems, suggesting the
  developer iterated on the design:

  OLD SYSTEM (GearPartSim / GearSimJoint):
    - GearPartSim handles everything for all gear types in one class
    - Contact detection, joint creation, joint update are all in
      GearPartSim's FixedUpdate
    - Simpler but less extensible
    - Uses static HasContact method on GearPartSim
    - Joints stored in Dictionary<GearPartSim, GearSimConnection>

  NEW SYSTEM (GearSim + GearController + GearScanner):
    - Polymorphic: each gear type has its own PartSim subclass
    - GearController handles scanning and joint management (separation
      of concerns)
    - GearScanner dispatch table supports all gear type combinations
      via a 2D array indexed by (GearType, GearType)
    - Joints stored in Dictionary<IGear, GearConn>
    - Cleaner architecture, easier to add new gear types

  The old system remains in the codebase but the new system is the
  one actively used (based on the interface hierarchy and scanner table).

13. SUPPORTING SYSTEMS
================================================================================

  GEAR SCANNER DISPATCH TABLE:
    A 2D array GearScanner[4,4] indexed by (GearType, GearType):
      [Spur, Spur]    → SpurSpurGearScanner
      [Spur, Bevel]   → SpurBevelGearScanner(reversed:true)
      [Spur, Linear]  → SpurLinearGearScanner(reversed:true)
      [Ring, Spur]    → RingSpurGearScanner(reversed:true)
      [Bevel, Bevel]  → BevelBevelGearScanner
      [Bevel, Linear] → BevelLinearGearScanner(reversed:true)
      [Linear, Linear] → LinearLinearGearScanner
      All others      → EmptyGearScanner (no contact possible)

  GEAR CONTROLLER:
    - Each gear has ONE GearController
    - Manages connections dictionary, neighbors list, toAdd/toRemove lists
    - Scan() called from SimPrePhysicsUpdate
    - Update() called from SimPhysicsUpdate
    - Separates "finding what to connect" from "updating connections"

  GEAR TRAIN (visual preview only):
    - GearTrain is a MonoBehaviour for design-time preview
    - gearTrain.Rotate(value) rotates self + recursively rotates
      all connections and stacks by the SAME angle
    - NO gear ratio is applied here — this is purely visual
    - GearTrainPower provides the animation driver (Update-based)

  PHYSICS SCENE:
    - ProjectSim manages a SEPARATE PhysicsScene for parts simulation
    - Physics.Simulate(physicsScene, step) is called manually
    - Configurable step rate (physicsSteps, ticksPerSecond)
    - Dynamic step rate via AnimationCurve physicsStepsOverFps

15. KEY INSIGHTS FOR RE-IMPLEMENTATION
================================================================================

  1. The ConfigurableJoint IS the gear constraint.
     Do not try to set transforms directly; let PhysX do the work.

  2. Anchors go on the CIRCUMFERENCE, not the center.
     This is the single most important geometric detail.

  3. Recalculate anchors EVERY physics step.
     The joint is re-anchored to the current contact teeth each frame.
     This creates the "rolling without slip" effect.

  4. Lock ALL 6 degrees of freedom.
     LinearLocked + AngularLocked on all axes.

  5. Overlap spheres are cheaper than custom collision detection.
     Physics.OverlapSphereNonAlloc with a reusable buffer.

  6. Deduplicate joint creation with instance ID ordering.
     Only the gear with the lower instance ID creates the joint between
     a pair. This prevents duplicate joints.

  7. Separate the scanning from the updating.
     Pre-physics: scan for neighbors, create/destroy joints.
     Post-physics: update anchors for surviving joints.

  8. Motor via Rigidbody torque or HingeJoint motor.
     Both work; AddTorque(ForceMode.VelocityChange) is simplest.

  9. The gear ratio is not your responsibility.
     Trust the physics solver. If anchors are correct, the ratio
     is correct.

 10. This approach works for any gear type pairing.
     Spur-spur, spur-bevel, ring-spur — as long as the scanner's
     CheckContact() produces a valid GearContact, the ConfigurableJoint
     handles the rest.
